## Feature 1: Fix FAQ Ticket Creation
**File:** `api/webhook/index.ts` — `handleFAQ()` function (line 333)

**What to change:**
- After Mistral AI responds, insert a `support_tickets` record with: `customer_phone`, `message` (user's question), `ai_response` (AI answer), `requires_human_intervention` (false), `status` ('auto_resolved')
- Update the Mistral system prompt to include: "If the question is completely unrelated to our services, respond with exactly: ESCALATE"
- When AI returns "ESCALATE", set `requires_human_intervention: true`, `status: 'open'`, and send a fallback message to the customer
- Also create tickets when there's no Mistral key (fallback path)

## Feature 2: Weekly Revenue Insight Cron
**New file:** `api/cron/weekly-revenue-insight.ts`

**Logic:**
1. Auth check (CRON_SECRET)
2. Query `service_cards` where `next_service_date` is in current month, with `customers(*)`
3. Query `reminder_responses` for non-responded reminders
4. Calculate: total due customers, total potential revenue (from service_details.totalCharge), non-responder count
5. Build Hindi message: "Namaste Sunil! Is mahine {count} customers due hain = ₹{amount} potential revenue. {nonResponders} ne last reminder ka reply nahi kiya. Follow-up bhejein?"
6. Send via OpenWA to merchant phone
7. Log to `cron_logs`

**vercel.json addition:**
```json
{ "path": "/api/cron/weekly-revenue-insight", "schedule": "0 9 * * 0" }
```

## Feature 3: Stock Alert Cron
**New file:** `api/cron/stock-alerts.ts`

**Logic:**
1. Auth check (CRON_SECRET)
2. Query `inventory` where `current_stock < minimum_threshold` and `merchant_id = MERCHANT_ID`
3. For each low item, check if unresolved alert already exists in `stock_alerts`
4. If not, insert new `stock_alerts` record
5. Build message listing all low-stock items with quantities
6. Send via OpenWA to merchant
7. Log to `cron_logs`

**vercel.json addition:**
```json
{ "path": "/api/cron/stock-alerts", "schedule": "0 8 * * *" }
```

## Feature 5: Idempotency Protection
**File:** `api/webhook/index.ts` — top of `processMessage()`

**Logic:**
1. Extract `idempotencyKey` from `payload.idempotencyKey`
2. Query Supabase: `SELECT 1 FROM webhook_idempotency WHERE key = $1`
3. If exists → return early (already processed)
4. After successful processing → INSERT into `webhook_idempotency (key, created_at)`
5. Periodic cleanup: DELETE WHERE `created_at < now() - interval '7 days'` (run in cron or on each request)

**Database migration needed:** Create `webhook_idempotency` table:
```sql
CREATE TABLE webhook_idempotency (
  key TEXT PRIMARY KEY,
  created_at TIMESTAMPTZ DEFAULT now()
);
```
